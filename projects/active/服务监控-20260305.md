## 🎯 项目概述

### 项目背景
为体现服务透明度，需要向用户实时展示各服务的运行状态、历史可用性、故障公告等信息，建立用户信任。

### 项目目标
- 目标1：实时展示各子域名服务状态
- 目标2：显示历史可用性数据（7天/30天/90天）
- 目标3：发布故障公告和恢复通知
- 目标4：提供状态订阅（邮件/RSS）

### 项目价值
- **用户价值**：实时了解服务状态，减少疑虑
- **业务价值**：透明化运营，建立信任
- **技术价值**：监控体系建设

## 📐 项目范围

### 核心职责
服务监控是 HamR 平台的**面向用户的状态透明页面**，负责：
1. **实时状态展示**：让用户了解各服务的当前运行状态
2. **历史可用性**：展示服务历史可用性数据，建立信任
3. **故障通报**：故障发生时及时发布公告，透明沟通
4. **状态订阅**：允许用户订阅状态更新通知

### 包含的功能

#### 1. 服务状态展示
- [ ] **监控服务清单**
  | 服务名称 | 域名 | 监控类型 |
  |---------|------|---------|
  | HamR 官网 | hamr.store | HTTP 200 检查 |
  | HamR 管家 | app.hamr.store | HTTP 200 + API 健康检查 |
  | 账号中心 | account.hamr.store | HTTP 200 + API 健康检查 |
  | 帮助中心 | help.hamr.store | HTTP 200 检查 |
  | 开发者门户 | hamr.top | HTTP 200 检查 |
  | 技术文档站 | docs.hamr.top | HTTP 200 检查 |
  | API 网关 | api.hamr.top | HTTP 200 + API 健康检查 |
  | 部署指南站 | deploy.hamr.top | HTTP 200 检查 |
  | 在线演示 | demo.hamr.top | HTTP 200 + API 健康检查 |
- [ ] **状态指示**
  - 🟢 正常运行（Operational）
  - 🟡 性能下降（Degraded Performance）
  - 🟠 部分中断（Partial Outage）
  - 🔴 完全中断（Major Outage）
  - 🔵 维护中（Under Maintenance）
- [ ] **状态页面布局**
  - 顶部总览（All Systems Operational / Issues Detected）
  - 各服务逐项状态
  - 当前/计划维护公告
  - 最近 7 天状态时间线

#### 2. 历史可用性数据
- [ ] **可用性统计**
  - 7 天可用性（天粒度柱状图）
  - 30 天可用性（天粒度）
  - 90 天可用性（周粒度）
  - 可用性百分比（如 99.95%）
- [ ] **响应时间**
  - 平均响应时间（各服务）
  - 响应时间趋势（折线图）
  - 峰值响应时间标记
- [ ] **事件历史**
  - 历史故障列表
  - 故障持续时间
  - 故障影响范围
  - 故障根因说明

#### 3. 故障公告管理
- [ ] **公告类型**
  - **故障公告**（Incident）
    - 发现时间
    - 影响范围
    - 当前状态（调查中/已识别/修复中/已解决）
    - 更新记录（时间线）
  - **维护公告**（Maintenance）
    - 计划维护时间
    - 影响范围
    - 预计持续时间
    - 维护完成确认
  - **通知公告**（Notice）
    - 服务升级通知
    - 功能变更通知
    - 安全更新通知
- [ ] **公告更新流程**
  ```
  发现问题 → 发布"调查中"公告
    → 确认原因 → 更新为"已识别"
      → 开始修复 → 更新为"修复中"
        → 修复完成 → 更新为"已解决"
          → 发布事后报告（RCA）
  ```

#### 4. 状态订阅
- [ ] **订阅方式**
  - 邮件订阅（输入邮箱，收到状态变更通知）
  - RSS 订阅（提供 RSS Feed）

- [ ] **通知规则**
  - 仅在状态变更时通知（避免频繁骚扰）
  - 可选择订阅全部/特定服务
  - 可随时取消订阅

  - 月度 SLA 达成情况
  - 目标 SLA（99.9%）
  - 实际 SLA 百分比

### 监控检查逻辑

#### 健康检查方式
```
【HTTP 检查】
- 发送 GET 请求到目标 URL
- 期望 HTTP 200 响应
- 超时阈值：10 秒
- 检查频率：每 5 分钟

【API 健康检查】
- 发送 GET 请求到 /health 端点
- 期望返回 {"status": "ok", "version": "1.0.0"}
- 检查数据库连接、Redis 连接、存储连接
- 超时阈值：15 秒
- 检查频率：每 3 分钟

【多点监控】
- 从多个地理位置发起检查（亚洲、欧洲、美洲）
- 多数检查点失败才判定为故障（避免误报）
```

### 服务边界与职责划分

#### 服务监控负责
✅ 面向用户的服务状态展示
✅ 故障和维护公告发布
✅ 历史可用性数据统计
✅ 状态变更通知

#### 服务监控不负责
❌ 内部运维监控和告警（由 Prometheus + Grafana 负责）
❌ 详细性能指标分析（由内部监控负责）
❌ 故障修复操作（由运维团队负责）
❌ 日志分析（由 ELK Stack 负责）

### 明确不包含
- 不包含：内部监控告警（使用 Prometheus）
- 不包含：详细性能指标（仅展示可用性和响应时间）
- 不包含：移动 App（仅 Web 页面）
- 不包含：自动故障修复

## 🛠 技术方案

### 技术栈
- **监控工具**：UptimeRobot / Uptime Kuma（自建）
- **前端**：React / Vue（静态页面）
- **托管**：Cloudflare Pages
- **域名**：status.hamr.top

### 架构设计
```
用户 → status.hamr.top（状态页面）
     ↓
   UptimeRobot API（监控数据）
     ↓
   各子域名服务（健康检查）
   ├─ hamr.store
   ├─ app.hamr.store
   ├─ account.hamr.store
   └─ ...
```
