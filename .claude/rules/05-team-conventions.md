# 团队协作约定

## 角色与职责

### 项目负责人
- 创建和维护项目主文档
- 定义项目目标和范围
- 跟踪项目进度和里程碑
- 及时更新项目状态

### 团队成员
- 贡献具体任务执行
- 更新相关文档（迭代、报告）
- 参与项目评审和回顾
- 提出改进建议

### AI 助手
- 辅助文档创建和格式化
- 快速检索项目信息
- 检查文档质量和一致性
- 提醒里程碑和截止日期

## 沟通约定

### 文档更新通知
```bash
# 重要更新需通知团队
git commit -m "update: 完成关键里程碑 - XXX"
# 发送通知：Slack/邮件/团队群组
```

### 评审流程
1. **项目立项**：需团队评审批准
2. **重大变更**：需讨论后更新文档
3. **项目归档**：需确认完成状态

### 会议记录
- 重要会议讨论应记录到相关项目文档
- 决策应在 Markdown 文档中明确标注
- 行动项应转化为任务列表

## 文档所有权

### 单一负责人原则
- 每个项目有且仅有一个负责人
- 负责人在项目注册表中明确标注
- 负责人负责文档的最终质量

### 协作编辑
- 其他成员可提出修改建议
- 通过 PR 或直接沟通方式
- 负责人审核后合并变更

## 命名约定

### 项目命名
- 使用清晰、描述性的名称
- 避免缩写和术语（除非团队熟知）
- 中文优先，英文作为补充

### 文件命名
- 遵循规范：`项目名称-YYYYMMDD.md`
- 日期使用创建日期，不轻易修改
- 归档时可添加后缀（如 `-completed`）

### 标签使用
- JSON 注册表中可使用 `tags` 字段
- 建议标签：`技术`, `产品`, `运营`, `基础设施`
- 保持标签一致性，避免同义词

## 工具使用规范

### Git 使用
```bash
# 提交前检查
git status
git diff

# 提交规范
git add .
git commit -m "类型: 简短描述"

# 推送前拉取
git pull --rebase
git push
```

### JSON 操作
```bash
# 验证 JSON 格式
jq . projects/registry/PROJ-XXX.json

# 查询项目
jq 'select(.status == "active")' projects/registry/*.json

# 更新字段（谨慎使用）
jq '.updated_date = "2026-03-04"' PROJ-XXX.json > tmp.json && mv tmp.json PROJ-XXX.json
```

### AI 助手协作
- 明确告知 AI 需要做什么
- 验证 AI 生成的内容
- 重要决策由人工确认
- 定期审查 AI 创建的文档

## 最佳实践

### DO ✅
1. **及时更新文档**
   - 项目有进展立即记录
   - 完成里程碑及时标记

2. **保持一致性**
   - 使用标准模板
   - 遵循命名约定
   - 同步更新 Markdown 和 JSON

3. **主动沟通**
   - 重大变更提前通知
   - 遇到问题及时讨论
   - 分享经验和教训

4. **定期回顾**
   - 每周查看项目进度
   - 每月归档完成项目
   - 每季度总结经验

### DON'T ❌
1. **不要跳过模板**
   - ❌ 手写文档不用模板
   - ✅ 从模板开始，保持一致

2. **不要延迟更新**
   - ❌ 等到项目结束再更新
   - ✅ 有进展立即记录

3. **不要单独工作**
   - ❌ 不通知就大改文档
   - ✅ 重要变更需讨论

4. **不要忽视质量**
   - ❌ 草率填写元信息
   - ✅ 认真检查文档质量

## 新成员入职

### 第一天
1. 阅读 `README.md` - 项目概述
2. 阅读 `AGENTS.md` - AI 助手使用指南
3. 阅读 `.claude/rules/` - 所有项目规则

### 第一周
1. 浏览现有项目文档（`projects/active/`）
2. 了解项目注册表使用方法
3. 尝试创建测试项目文档
4. 与团队成员讨论疑问

### 持续学习
- 参与项目评审和回顾
- 观察其他成员的文档风格
- 提出改进建议
