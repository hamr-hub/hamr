## 1. 响应级别定义

| 级别 | 名称 | 影响范围 | 响应时间 | 示例 |
|------|------|---------|---------|------|
| P0 | 紧急 | 全站不可用 / 数据安全 | < 15 分钟 | 服务器宕机、数据泄露 |
| P1 | 严重 | 核心功能不可用 | < 30 分钟 | API 网关故障、数据库连接失败 |
| P2 | 一般 | 部分功能异常 | < 2 小时 | 单个子站点无法访问、慢查询 |
| P3 | 轻微 | 非关键问题 | < 24 小时 | 样式错误、非核心页面404 |

## 2. 响应流程

```
告警触发 / 用户上报
    │
    ▼
[1] 值班人员确认（< 5 分钟）
    │
    ▼
[2] 评估影响范围，确定级别（< 10 分钟）
    │
    ├── P0/P1 → 立即启动应急小组
    └── P2/P3 → 创建工单，按优先级处理
    │
    ▼
[3] 执行对应预案
    │
    ▼
[4] 验证恢复
    │
    ▼
[5] 发布公告（status.hamr.top）
    │
    ▼
[6] 事后复盘（48 小时内完成 RCA）
```

## 3. 预案详情

### 3.1 服务器宕机（P0）

**现象**：所有站点无法访问，Prometheus 告警触发

**排查步骤**：
1. SSH 连接服务器：`ssh root@43.133.224.11`
2. 如果 SSH 不通，登录阿里云控制台检查实例状态
3. 检查系统负载：`top`, `free -h`, `df -h`
4. 检查 Docker 状态：`docker ps -a`

**恢复操作**：
```bash
# 场景1: Docker 服务异常
systemctl restart docker
cd /opt/hamr/proxy && docker compose up -d

# 场景2: 磁盘满
docker system prune -af
journalctl --vacuum-size=100M

# 场景3: 内存不足
# 识别高内存进程并重启
docker stats --no-stream
docker restart <container_name>

# 场景4: 服务器需要重启
reboot
# 等待重启后
cd /opt/hamr/proxy && docker compose up -d
```

### 3.2 数据库连接失败（P1）

**现象**：API 返回 500，日志显示数据库连接错误

**排查步骤**：
1. 检查 PostgreSQL 容器状态：`docker ps | grep postgres`
2. 检查数据库日志：`docker logs hamr-postgres --tail=50`
3. 检查连接数：`docker exec hamr-postgres psql -U hamr -c "SELECT count(*) FROM pg_stat_activity;"`
4. 检查磁盘空间：`docker exec hamr-postgres df -h /var/lib/postgresql/data`

**恢复操作**：
```bash
# 场景1: 容器崩溃
docker restart hamr-postgres

# 场景2: 连接数耗尽
docker exec hamr-postgres psql -U hamr -c "SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state = 'idle' AND query_start < now() - interval '10 minutes';"

# 场景3: 数据损坏 → 从备份恢复
/opt/hamr/scripts/restore.sh postgres /opt/hamr/backups/postgres/latest.sql.gz
```

### 3.3 Nginx 代理故障（P1）

**现象**：部分或全部站点返回 502/503

**排查步骤**：
1. 检查 Nginx 容器：`docker logs nginx-proxy --tail=50`
2. 检查后端容器状态：`docker compose ps`
3. 验证 Nginx 配置：`docker exec nginx-proxy nginx -t`

**恢复操作**：
```bash
# 场景1: 配置错误
docker exec nginx-proxy nginx -t
# 修复配置后
docker exec nginx-proxy nginx -s reload

# 场景2: 后端容器挂了
docker compose up -d

# 场景3: 需要完全重建
docker compose down
docker compose up -d
```

### 3.4 SSL 证书过期（P2）

**现象**：浏览器提示证书不安全

**恢复操作**：
```bash
# 手动续期
certbot renew --force-renewal

# 重载 Nginx
docker exec nginx-proxy nginx -s reload
```

### 3.5 DDoS 攻击（P0）

**现象**：服务响应极慢或不可用，流量异常激增

**应急操作**：
1. 确认攻击类型（查看 Nginx access log）
2. 开启阿里云 DDoS 防护
3. 阿里云 WAF 紧急规则：封禁异常 IP 段
4. 必要时切换 DNS 到阿里云 CDN 防护节点
5. 通知相关团队

```bash
# 查看高频 IP
docker exec nginx-proxy sh -c "awk '{print \$1}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head -20"

# 临时封禁 IP（在 Nginx 配置中添加）
# deny 123.45.67.89;
```

### 3.6 数据泄露（P0）

**现象**：发现未授权数据访问

**应急操作**：
1. **立即**：隔离受影响系统（断网/关闭端口）
2. 保全证据（日志快照、网络抓包）
3. 评估泄露范围（哪些数据、多少用户）
4. 轮换所有 API Key 和数据库密码
5. 通知受影响用户
6. 向监管部门报告（如适用）

```bash
# 轮换数据库密码
docker exec hamr-postgres psql -U postgres -c "ALTER USER hamr PASSWORD 'new_secure_password';"

# 更新 .env 文件中的密码
# 重启所有依赖数据库的服务
docker compose restart
```

### 3.7 DNS 劫持（P0）

**现象**：域名解析到非预期 IP

**排查**：
```bash
dig hamr.store +short
dig hamr.top +short
# 对比是否为 43.133.224.11
```

**应急操作**：
1. 登录阿里云 DNS 控制台检查记录
2. 如果记录被篡改，立即修复
3. 修改阿里云账号密码，开启 MFA
4. 启用 DNSSEC
5. 检查是否有未授权的 API Key
